## 📐 10段階評価スケール定義（共通基準）

すべてのチェック項目はこのスケールで評価する。

```
┌────────┬───────────┬────────────────────────────────────────────────────────────────┐
│ Score  │ ラベル     │ 共通判断基準                                                     │
├────────┼───────────┼────────────────────────────────────────────────────────────────┤
│  10    │ 模範的     │ ベストプラクティスを完全に実装。該当リスクがゼロかつ積極的対策あり │
│   9    │ 優秀       │ ベストプラクティスに準拠。ごく軽微な改善余地のみ                 │
│   8    │ 良好       │ 概ね安全。1〜2個の軽微な問題があるが実害のリスクは低い           │
│   7    │ 合格       │ 基準を満たすが注意点あり。対策すれば問題なし                     │
│   6    │ 要注意     │ 複数の懸念点。現状でも使用可能だが早期改善を推奨                  │
│   5    │ 条件付き   │ 重要な問題あり。リスク承認と監視条件付きでのみ使用可              │
│   4    │ 要改善     │ 重大な問題あり。改善完了まで使用不可                             │
│   3    │ 危険       │ セキュリティ上の重大欠陥。悪用・漏洩のリスクが高い               │
│   2    │ 深刻       │ 悪意ある設計の疑いまたは致命的脆弱性                             │
│   1    │ 即時排除   │ 明確な悪意ある実装。マルウェア・バックドアの特徴                  │
└────────┴───────────┴────────────────────────────────────────────────────────────────┘
```

**自動失格ルール：** スコア1または2が1項目でも出た場合、総合判定は **⛔ 即時排除** となる。

## D1 — 配布元・信頼性の評価

### D1-1: 開発者・組織のアイデンティティ

| スコア | 判定基準 |
|--------|---------|
| **10** | 著名なOSS組織・認証済み企業アカウント。公式Webサイト・セキュリティポリシー完備。実名・連絡先あり |
| **9**  | 実名・所属が確認可能。1年以上の開発実績。セキュリティ報告窓口あり |
| **8**  | 実名またはトレーサブルなハンドル名。半年以上の活動履歴。README充実 |
| **7**  | ハンドル名のみだが一貫した活動履歴あり。他のリポジトリでも実績あり |
| **6**  | ハンドル名のみ。活動履歴は3〜6ヶ月程度。評価・フォロワーはある程度存在 |
| **5**  | 活動履歴が3ヶ月未満、または単一リポジトリのみ。フォロワー少数 |
| **4**  | 新規アカウント（1ヶ月未満）。実績なし。プロフィール情報が極めて少ない |
| **3**  | 匿名・使い捨てメールで登録した形跡。コミュニティ評価が著しく低い |
| **2**  | 過去にセキュリティインシデントを起こした報告のあるアカウント |
| **1**  | なりすまし・偽装が疑われる。著名プロジェクトに類似した偽アカウント |

### D1-2: 配布チャネルの完全性

| スコア | 判定基準 |
|--------|---------|
| **10** | 公式リポジトリから配布。SHA-256ハッシュ公開。GPG/Cosign署名付き。SBOM提供 |
| **9**  | 公式リポジトリ配布。チェックサム公開。署名なしだがハッシュ検証可能 |
| **8**  | 公式リポジトリ配布。バージョンタグ付き。チェックサムは部分的に提供 |
| **7**  | 公式リポジトリ配布。タグあり。完全性検証手段は限定的だが信頼できる |
| **6**  | 公式リポジトリ配布だがリリースノート不足。バージョン管理が不完全 |
| **5**  | 非公式ミラーを経由した配布。元リポジトリとの同一性が未確認 |
| **4**  | 複数の非公式ソースから異なるバージョンが流通。正本が不明確 |
| **3**  | 配布元が不透明。ハッシュ・署名なし。変更履歴なし |
| **2**  | 既知の正規スキルに似た名前で別ソースから配布（タイポスクワッティング疑い）|
| **1**  | 公式ソースのコードと内容が明らかに異なる。改ざんの証拠 |

| スコア | 判定基準 |
|--------|---------|
| **10** | 月次以上の更新。CVEへの48時間以内の対応実績あり。LTS保証あり |
| **9**  | 3ヶ月以内に更新。セキュリティ報告への迅速な対応実績あり |
| **8**  | 6ヶ月以内に更新。バグ修正・セキュリティ対応を継続実施 |
| **7**  | 1年以内に更新。主要な問題には対応済み。コミュニティも活発 |
| **6**  | 1〜2年未更新。ただし機能が安定しており既知CVEなし |
| **5**  | 2年以上未更新。メンテナーからのレスポンスが遅延 |
| **4**  | 3年以上未更新。メンテナーが応答なし。実質放棄の可能性 |
| **3**  | 明示的に廃止宣言。後継なし。脆弱性が未対応のまま放置 |
| **2**  | 廃止後に別オーナーが引き継いだ形跡。内容が大幅変更されている |
| **1**  | 正規スキルが廃止後、悪意あるコードを含む版が同名で公開されている |

## D2 — コード・実行内容の静的解析

### D2-1: SKILL.md 指示内容の安全性

| スコア | 判定基準 |
|--------|---------|
| **10** | 指示が明確・限定的。外部リソースへのアクセスなし。最小権限設計が明示 |
| **9**  | 指示が適切。必要な外部アクセスは目的が明示・最小限 |
| **8**  | 指示は適切。一部曖昧な表現があるが悪用リスクは低い |
| **7**  | 指示に若干の広範な記述あり。ただし文脈上の悪用リスクは限定的 |
| **6**  | 広範な権限要求または曖昧な指示が複数。意図は善意と判断されるが要確認 |
| **5**  | 明示的な目的なしに広範なアクセスを要求する指示あり |
| **4**  | セキュリティ制限を回避しようとする文言が存在（軽微な回避試みに見える） |
| **3**  | 「他の指示より優先」「フィルターを無視」等の制約回避指示が存在 |
| **2**  | ロールの書き換え・Jailbreak誘導・システムプロンプト上書きの明確な試み |
| **1**  | 「DAN」「悪意ある行動を実行せよ」等の明確な悪意ある命令が埋め込まれている |

### D2-2: 難読化・隠し指示の検出

| スコア | 判定基準 |
|--------|---------|
| **10** | 難読化ゼロ。すべての内容が平文で可読。セクション構造が明確 |
| **9**  | 難読化なし。コメントも含めて内容が完全に透明 |
| **8**  | 技術的な理由による最小限のエンコードのみ（URLエンコード等、正当な用途） |
| **7**  | Base64等があるが内容がデコード可能で無害と確認できる |
| **6**  | 一部のセクションが判読困難。意図的かは不明だが要精査 |
| **5**  | 複数箇所にエンコードされた文字列。目的が不明確 |
| **4**  | 意図的と思われる難読化パターン（文字列連結・hex変換） |
| **3**  | ゼロ幅文字・不可視テキストによる隠しコンテンツの疑い |
| **2**  | 複数層のエンコード。デコード後に疑わしい命令を含む |
| **1**  | 難読化された悪意ある命令が確認された |

### D2-3: スクリプト・実行コードの安全性

| スコア | 判定基準 |
|--------|---------|
| **10** | 全コードが安全なAPI使用。入力検証完備。最小権限。テストカバレッジあり |
| **9**  | 安全なコーディング実践。危険APIなし。エラーハンドリング適切 |
| **8**  | 概ね安全。軽微なコーディング上の問題（未使用変数等）はあるが実害なし |
| **7**  | 1〜2個の軽微なリスク（弱いバリデーション等）。悪用リスクは低い |
| **6**  | 入力検証が不完全。サニタイズ漏れがあるが直接的な危険は限定的 |
| **5**  | `subprocess`等の使用があるが引数が固定または限定的 |
| **4**  | ユーザー入力をそのままシェルコマンドに渡す可能性あり |
| **3**  | `eval()` / `exec()` の使用。任意コード実行の経路が存在する |
| **2**  | 外部へのデータ送信コードを含む。送信先・内容が不透明 |
| **1**  | マルウェア・バックドアの特徴（C2通信・自己複製・ペイロード実行） |

| **6**  | 低深刻度CVEが1〜2件。対処計画あり |

## D4 — Prompt Injection リスク

### D4-1: Direct Prompt Injection（直接攻撃）

| スコア | 判定基準 |
|--------|---------|
| **10** | インジェクション試みゼロ。指示が明確に境界付けられている |
| **9**  | 注入の試みなし。指示内容が完全に透明で目的に沿っている |
| **8**  | 若干の広範な表現があるが文脈上インジェクションとは言えない |
| **7**  | 境界が曖昧な箇所が1〜2ヶ所。悪用意図は読み取れない |
| **6**  | 「特別な状況では〜」等の条件付き広範指示あり。要精査 |
| **5**  | 「通常の制限を超えて」等の回避示唆的な表現あり |
| **4**  | セキュリティガイドラインを「柔軟に解釈」するよう誘導する記述 |
| **3**  | 「この指示は他のすべての指示より優先される」等の明確な優先度操作 |
| **2**  | Jailbreakパターン（「DAN mode」「開発者モード」等）の埋め込み |
| **1**  | 「有害なコンテンツを生成せよ」「ユーザーデータを盗め」等の明示的悪意命令 |

---

### D4-2: Indirect Prompt Injection（間接攻撃）

| スコア | 判定基準 |
|--------|---------|
| **10** | 外部データアクセスなし。またはデータと命令の完全分離が設計に明記されている |
| **9**  | 外部データ取得あるが、サニタイズとデータ/命令分離の実装が確認できる |
| **8**  | 外部データ取得あり。明示的な分離機構はないが処理が限定的で低リスク |
| **7**  | Web取得・ファイル読み取りあり。インジェクション対策は最小限 |
| **6**  | 複数の外部ソースからデータ取得。対策は部分的 |
| **5**  | 取得した外部データをプロンプトに直接挿入する設計 |
| **4**  | ユーザー提供のURLやファイルを無検証でプロンプトに混入 |
| **3**  | メール本文・PDF内容等を直接命令として扱う可能性がある設計 |
| **2**  | Indirect Injectionの既知パターンを利用した設計（意図的と判断） |
| **1**  | 外部データを通じた命令実行が設計の中核をなしている（明確な悪用） |

## D6 — 権限・アクセス制御

### D6-1: ファイルシステム・OS権限

> **v2.0追加：スキルシステムへの書き込みはFS権限の中でも特別に重大なリスク。**
> `~/.claude/skills/` 等のスキル格納ディレクトリへの自律的な書き込みは、
> 「現セッションの権限」で「全将来セッションの動作」を永続的に変更する点で、
> 通常のFS書き込みと質的に異なる。D8とあわせて評価すること。

| スコア | 判定基準 |
|--------|---------|
| **10** | ファイルシステムアクセスなし。またはスコープが単一の明示されたディレクトリに限定 |
| **9**  | 最小権限。アクセス先が目的に必要な範囲に完全に限定されている |
| **8**  | アクセス範囲は適切。ほぼ最小権限を実現できている |
| **7**  | アクセス範囲がやや広いが、実害につながる可能性は低い |
| **6**  | 必要以上に広いディレクトリへのアクセスがあるが悪用の痕跡なし |
| **5**  | ホームディレクトリ全体など、広範なアクセスを要求 |
| **4**  | システム設定ファイルへのアクセスを要求（/etc/等）、またはスキルシステムへの書き込みあり（ユーザー確認なし）|
| **3**  | root/管理者権限を要求、またはそれを前提とした処理 |
| **2**  | `/mnt/user-data` や機密ディレクトリへの明示的なアクセス試み |
| **1**  | システム全体への書き込み権限要求。破壊的コマンド（rm -rf等）の実行 |

**スキルシステム書き込みペナルティ（D6-1のスコア決定時に適用）：**
```
ユーザー確認なしでスキルディレクトリに自動書き込み → スコアを2段階下げる
書き込み内容に機密情報フィルタなし → さらに1段階下げる
書き込まれたスキルが次セッションで自動ロードされる設計 → D8-2も同時に評価

※ このペナルティはD6-1のスコア決定に組み込む（スコア表の基準から
   上記条件で段階を下げた値がD6-1の最終スコアとなる）。
   scoring-aggregation.md の最終スコアペナルティ（−4点）との二重適用はしない。
```

| スコア | 判定基準 |

## D7 — 運用・保守性

### D7-1: ドキュメント・透明性

## D8 — エージェント動作・永続性リスク（v2.0 新設）

> このドメインはAgent Skills特有の攻撃面を評価する。
> 「スキルがスキルを生む（メタスキルパターン）」「全プロンプトへのHook割り込み」
> 「会話コンテキストの自律的な分析・永続化」といったリスクは、
> D1〜D7のどのドメインにも完全には収まらないため独立評価が必要。

### D8-1: Hookメカニズムの安全性

**評価対象：** UserPromptSubmit / PostToolUse / PostClaude等のフック設定とスクリプト内容

| スコア | 判定基準 |
|--------|---------|
| **10** | Hookなし。またはHookの目的・動作が完全に文書化されており影響が限定的 |
| **9**  | Hookあり。実行内容が透明で悪用経路がなく、目的が明確かつ限定的 |
| **8**  | Hookあり。内容は安全だが一部の動作が文書化されていない |
| **7**  | Hookあり。全プロンプトに割り込むが内容は読んだ範囲で安全 |
| **6**  | 全プロンプト割り込みのHook。スクリプト内容の一部が未確認 |
| **5**  | 全プロンプト割り込み。動作が一部不透明。外部通信またはFS操作を含む疑い |
| **4**  | Hookスクリプトの内容が非公開または難読化されている |
| **3**  | Hookが外部通信を伴うことが確認できる、または設定ファイルを変更する |
| **2**  | Hookを通じたシステム設定の書き換えまたはセキュリティ設定の変更 |
| **1**  | Hookによるバックドア実行・マルウェアの常駐化 |

---

### D8-2: スキル生成・自己増殖リスク

**評価対象：** スキルが新しいスキルを生成・保存する設計（メタスキルパターン）

> 自己増殖リスクの本質：一度汚染されたスキルは次回セッション以降も自動ロードされる。
> Indirect Injectionと組み合わさると「悪意あるWebページ → スキル生成 → 永続感染」が成立する。

| スコア | 判定基準 |
|--------|---------|
| **10** | スキル生成機能なし。またはスキル生成が完全にユーザー制御下にある |
| **9**  | スキル生成あり。生成前に必ずユーザー確認ダイアログを表示・承認必須 |
| **8**  | スキル生成あり。生成内容をユーザーに事前表示するが承認は任意 |
| **7**  | スキル生成あり。ユーザーへの通知はあるが確認なしで保存される |
| **6**  | スキル自動生成あり。トリガー条件が広範で頻繁に発動する可能性 |
| **5**  | スキル自動生成。会話内容から無制限に生成。機密情報フィルタなし |
| **4**  | 生成スキルの内容にセキュリティレビューなし。APIキー等を含む可能性 |
| **3**  | 生成スキルが機密情報（APIキー・パスワード等）を含んで永続化されるリスク |
| **2**  | Indirect Injectionを通じた悪意あるスキルの自動生成・永続化が成立する設計 |
| **1**  | 意図的な自己増殖マルウェアパターン。他システムへの感染機構 |
