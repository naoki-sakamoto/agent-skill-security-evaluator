# Agent Skills セキュリティ評価スキル v2.0

## 概要

第三者開発のAgent Skillsを受け取り、**8つのセキュリティドメイン × 各チェック項目を10段階評価**し、
構造化された評価レポートを出力する。

v2.0の主な変更点：
- **D8「エージェント動作・永続性リスク」を新設**（Hook機構・スキル自己生成パターンに対応）
- GitHub評価の情報収集手順を明文化
- スコア計算ステップの透明化（show-work義務化）
- 複数リポジトリ同時評価モードを追加
- D6にエージェントFS書き込みの永続感染リスク判定軸を追加

---

## 評価実行フロー

### Step 1: 評価対象の受け取りと前処理

評価対象として以下のいずれかを受け取る：
- アップロードされたSKILL.mdファイル
- GitHubリポジトリURL
- 会話内に貼り付けられたコード・設定テキスト
- スクリプトファイル（.py / .js / .sh等）

#### 通常ファイル・コードの場合

```
1. ファイルの文字コードと構造を確認
2. YAML frontmatterの存在と形式を確認
3. 添付ファイルが複数ある場合は依存関係を整理
4. 外部URLや参照リソースがあれば一覧化
5. allowed-tools フィールドがあれば許可ツール一覧を記録
```

#### GitHubリポジトリURLを受け取った場合の情報収集手順

**取得ステップ（上から順に実行）：**

1. **リポジトリトップページをfetch**
   - Stars数・Forks数・コントリビューター数・言語構成・最終更新・Issue数を記録

2. **web_searchで内容を補完**（直接fetchできないパスに有効）
   ```
   クエリ例：site:github.com owner/repo SKILL.md
   クエリ例：owner/repo scripts security analysis
   クエリ例：owner reponame skill content playbooks agentskills
   ```
   playbooks.com / agentskills.so 等のミラーサイトにSKILL.md全文が掲載されていることが多い

3. **スクリプトが確認できない場合**
   - 評価信頼度を ⭐⭐⭐ 中に下げる
   - D2-3・D6・D8 の評価で「情報不足による推定評価」と明記する

4. **複数スキルを含むリポジトリの場合**（100+スキルのマーケットプレイス等）
   - スキル数が10以上の場合は **5件のランダムサンプリング評価** を実施
   - サンプリングした旨・対象ファイル名をレポートに明記

#### 複数リポジトリを同時に評価する場合

```
1. まず全リポジトリのトップページを並行fetchして概要を把握
2. 各リポジトリを独立して全18項目評価（スコアの相対比較をしない）
3. 全リポジトリ評価完了後に比較サマリーテーブルを出力
4. 判定が異なる場合はその理由を明記する

⚠️ 重要：A のスコアが高いからといって B のスコアを
   相対的に低く調整しないこと。絶対基準で評価する。
```

---

### Step 2: 8ドメイン評価の実行

詳細な評価基準は `references/evaluation-rubric.md` を参照。

| # | ドメイン | 重み係数 | チェック項目数 |
|---|----------|----------|--------------|
| D1 | 配布元・信頼性 | 1.0× | 3項目 |
| D2 | コード・実行内容の静的解析 | 1.5× | 3項目 |
| D3 | 依存関係・サプライチェーン | 1.2× | 2項目 |
| D4 | Prompt Injection リスク | 1.5× | 2項目 |
| D5 | データ・プライバシー | 1.3× | 2項目 |
| D6 | 権限・アクセス制御 | 1.2× | 2項目 |
| D7 | 運用・保守性 | 0.8× | 2項目 |
| D8 | エージェント動作・永続性リスク | 1.4× | 2項目 |

> **重要：** D2・D4・D8はAgent Skills固有の攻撃ベクターを含むため重み係数が高い。
> いずれかの項目で**スコア2以下**が出た場合、総合判定は自動的に「⛔ 即時排除」となる。

---

### Step 3: スコアの集計（show-work 義務）

`references/scoring-aggregation.md` の計算式に従い、**以下のステップを省略せず明示する：**

```
【Step 3-1】 18項目スコアを一覧表示（省略不可）
  D1-1: X  D1-2: X  D1-3: X
  D2-1: X  D2-2: X  D2-3: X
  D3-1: X  D3-2: X
  D4-1: X  D4-2: X
  D5-1: X  D5-2: X
  D6-1: X  D6-2: X
  D7-1: X  D7-2: X
  D8-1: X  D8-2: X

【Step 3-2】 自動失格チェック（以下のいずれかで即座に ⛔ 宣言・計算終了）
  □ いずれかの項目のスコアが 1 または 2
  □ D4（Prompt Injection）ドメインの平均スコアが 4 未満
  □ D2-3（スクリプト安全性）のスコアが 3 以下
  □ D5-1（機密データ処理）のスコアが 2 以下
  □ D8-1（Hook安全性）のスコアが 3 以下
  □ D8-2（スキル自己生成）のスコアが 2 以下

【Step 3-3】 ドメイン平均を計算（8ドメイン分を明示）
  D1平均 = (D1-1 + D1-2 + D1-3) ÷ 3 = XX.X
  ...（全ドメイン記載）

【Step 3-4】 加重スコアを計算（8ドメイン分を明示）
  D1加重 = D1平均 × 3.0 = XX.X
  ...（全ドメイン記載）

【Step 3-5】 加重合計 ÷ 223 × 100 = 総合スコア（100点換算）

【Step 3-6】 ペナルティ適用 → 最終スコア確定

【Step 3-7】 最終判定を宣言

【Step 3-8】 Critical/High項目の集計（スコア1-5の項目を証拠とともに記録）
```

> 💡 計算ステップを省略しないこと。透明性がスキルの信頼性そのものを高める。

---

### Step 4: 評価レポートの生成

`references/report-template.md` のフォーマットに従い、以下を含むレポートを出力：

```
[構造]
1. エグゼクティブサマリー（3-5行）
2. 総合スコアと判定（ペナルティ内訳含む）
3. ドメイン別スコアカード（D1〜D8）
4. 全18項目詳細スコア（ドメインごとに表形式）
5. Critical/High項目の詳細（証拠・発見箇所付き）
6. 改善提案（Priority 1〜4）
7. 最終推奨アクション
8. 評価メタデータ（信頼度・未評価項目・次回評価日）
```

## 10段階評価スケール（概要）

詳細な判定基準は `references/evaluation-rubric.md` § 評価スケール定義 を参照。

| スコア | ラベル | 意味 |
|--------|--------|------|
| **10** | 模範的 | セキュリティベストプラクティスを完全に満たす |
| **9** | 優秀 | 軽微な改善余地があるが全体的に安全 |
| **8** | 良好 | 小さな問題があるが許容範囲内 |
| **7** | 合格 | いくつかの懸念点があるが管理可能 |
| **6** | 要注意 | 複数の懸念点があり対策が必要 |
| **5** | 条件付き | 重要な問題があり改善なしでの使用は非推奨 |
| **4** | 要改善 | 重大な問題があり現状では使用不可 |
| **3** | 危険 | セキュリティ上の重大な欠陥が存在 |
| **2** | 深刻 | 悪意ある設計または致命的な脆弱性の疑い |
| **1** | 即時排除 | 明確な悪意ある実装またはマルウェアの特徴 |

**自動失格：** スコア1または2が1項目でも出た場合 → **⛔ 即時排除**（総合スコア計算なし）

## 注意事項

- 静的解析のみ実施。動的実行・サンドボックステストは別途必要
- 評価結果はあくまで参考。最終判断は組織のセキュリティポリシーに従うこと
- スコアは提供された情報に基づく推定値であり、完全な保証を意味しない
- 複数スキルを持つリポジトリの評価信頼度はサンプリング数に依存する
